The Interface Just Split in Two

Building a prompt-based personal operating system where the agent handles input and the screen handles truth.

For most of software history, one screen did two different jobs. It collected input from the human, then presented output back to the human. AI agents make that coupling optional.

A personal finance app asks you to choose a category, enter an amount, pick an account, and submit a form. A time tracker asks you to translate a messy day into the fields its product manager anticipated. A task manager asks you to maintain its ontology before it can help you use it.

The friction is not the click. The friction is the translation. The person has to convert life into the application's schema before the application can do anything useful.

The old application used the page as the universal interface. The new application can split the interface in two: conversation for input; a visual surface for observation and confirmation.

1. The interface just split in two

I am using Agent-Mediated Software for a strict version of this pattern:

InputThe human describes intent in natural language.
TranslationThe agent resolves that intent into typed operations.
StateDeterministic code validates and stores the result.
OutputThe visual interface shows state and asks for confirmation when a change is durable.

This is not “no UI.” The screen still matters. It is simply released from being the only way into the system. It can concentrate on the things screens do well: comparison, legibility, overview, and consent.

The ingredients have clear predecessors. Agent-native software treats the agent and UI as partners over shared state. CLI-Anything argues that agents need structured, deterministic control surfaces instead of brittle pixel imitation. The narrower experiment here is the division of labor: the agent is the normal write path; the portal is the observe-and-confirm path.

2. Prompt-based does not mean prompt-only

The tempting slogan is “personal software should become prompt-based instead of code-based.” The useful version is more precise.

Code moves inward. It owns schemas, validation, balances, time arithmetic, idempotency, and permission boundaries. Prompts move outward. They become the layer where a person expresses what happened, changes how the system models life, and asks the agent to adapt the product.

That separation matters. I do not want financial history stored as chat. I want a strict ledger. I simply do not want to fill out the ledger by hand.

It also changes who gets to shape the product. A traditional app can only present the categories, views, and workflows its product team predicted. An agent-mediated personal system can start from a small method and let each user ask for the categories, currencies, rankings, and views that make their own life legible. The underlying code remains constrained; the model around the person can evolve.

3. A small test: Today, Time, and Money

I extracted the idea into Agentic Life OS, a local-first open-source project with only three modules:

Today

One daily focus and a ranked list of actionables. These changes are reversible, so the agent can write them directly.

Time

A 168-hour weekly budget, a physical-time ledger, allocation credit, and consumption rankings.

Money

Accounts, a monthly plan, and a transaction ledger with preview and explicit commit.

Today is intentionally small. A portal should not become a second life to maintain. It should answer one question quickly: what matters now?

Time is where a personal model becomes more interesting. One physical hour can serve two intentions. A train ride is still one hour of clock time, even if it also advances a learning budget. The system therefore keeps clock time and allocation credit separate. Its newest view ranks where the week physically went, first by meaningful category and then by the individual budget items underneath, without double-counting overlap.

Money is stricter. Transfers must not look like income. Refunds must not look like spending. Retrying the same agent operation must not create a duplicate. The default demo uses USD, while the schema carries a currency code so another currency can be added without pretending exchange rates are simple.

4. The agent does not get a blank check

The operating loop is deliberately boring:

  1. Observe. Read Today plus the current Time and Money summaries.
  2. Propose. Translate raw notes into normalized operations.
  3. Confirm. Show the person exactly what will change.
  4. Commit. Apply the accepted proposal atomically.

Reversible tasks and focus changes can move immediately. Durable Time and Money entries cannot. The agent prepares a proposal; the person approves the normalized version; only then does the ledger change.

This is the part a pure chat interface usually misses. Natural language is flexible, but it is not a substitute for transaction semantics. The agent stays probabilistic at the edge. The ledger stays deterministic at the center.

5. Personalization becomes model revision

Most personal software calls customization a settings page. An agent-mediated system can go further: it can treat the user's model as a hypothesis.

After a completed week or month, the agent can inspect the evidence and propose a revision:

  • a recurring unbudgeted activity may deserve its own category;
  • a repeated overshoot may reveal a bad plan, a classification mistake, or behavior the person actually wants to change;
  • an unused item may need to shrink, merge, disappear, or remain protected on purpose;
  • a user can ask for a new currency or a new presentation without waiting for a product roadmap.

The goal is not to make every bar green. It is to make the model more truthful. And the adaptation is never silent: the agent shows the evidence and waits for human confirmation before changing the budget itself.

6. Run it locally, then make it reachable

The application contains no model and no AI credentials. Bring Codex, Claude, OpenClaw, or another agent that can call an HTTP API or CLI. The default demo is synthetic and runs locally with one command:

git clone https://github.com/qilinxiaoxiang/agentic-life-os.git
cd agentic-life-os
docker compose up --build

Then open http://127.0.0.1:5050. Start with the demo, delete it when the model makes sense, and let an agent initialize a clean Time and Money budget from the facts you provide.

Phone access should not require turning a personal ledger into a public website. The repository now includes a short Tailscale setup: install it on the computer and phone, keep Life OS bound to localhost, and use Tailscale Serve to expose it only inside your private network. Do not use Funnel; this app has no application-level login.

7. What this experiment does not claim

Forms remain better when the schema is stable, speed matters, and ambiguity is expensive. Agents add latency, probabilistic interpretation, and a new dependency. A tax filing or wire transfer should not become casual because natural language feels easy.

Nor is the phrase “agent-mediated” itself new. The contribution I care about is the boundary tested in working software: agent-only normal writes, typed state, an observe-and-confirm portal, and human-governed adaptation.

The next personal application may not be a finished product that asks everyone to live inside the same schema. It may be a small, strict operating core that each person reshapes through an agent — while the screen simply shows the truth.
