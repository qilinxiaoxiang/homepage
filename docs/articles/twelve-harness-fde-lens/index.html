What Twelve AI Coding Harnesses Miss for Forward-Deployed Work

I read the source of twelve popular AI coding harnesses before installing any of them. None is designed for the Forward-Deployed Engineering client engagement. That gap is the opportunity.

I was about to install one of the popular AI coding harnesses. Instead I opened the source. Twelve frameworks later, I have not installed any of them, and I am clearer about what is missing.

The clearest finding is uncomfortable for the ecosystem. Of the twelve frameworks I read, with a combined star count north of half a million, none is designed for the work I actually do as a Forward-Deployed AI Engineer. They assume the author is the client and the reviewer. That assumption removes the four hardest parts of client engagement and leaves a tidy product behind.

This is also a market signal. If every popular harness solves the easy half of the work, the hard half is unclaimed.

The hard half of FDE delivery is not code generation. It is spec sign-off, architecture decision records, human-in-the-loop gates, and an audit trail a client can read.

1. The question that triggered this

A client engagement does not start with "what would you like to automate." It starts with "here is the contract, here is the deadline, here is the person who has to approve every irreversible change." The work is not measured by lines of code shipped. It is measured by whether the client can read the audit trail in three months and still trust it.

So when I sat down to pick a harness for that work, the question was simple. Does this framework treat the engineer and the reviewer as the same person? If yes, it solves the wrong problem. I read all twelve. The answer was yes, eleven times. The twelfth, Traycer Bart, hides the answer behind a closed-source LLM judge, which is the same failure in different clothes.

This article is what I learned. It is also a companion to my prior post on Forward-Deployed Engineering field method, which covers the discovery side of the same job.

2. What "fit for FDE" actually means

A harness fits FDE when it treats the four hardest parts of client work as first-class.

Spec sign-off

The PRD lives in git, the client approves it on a pull request, and no agent runs until that approval exists. The spec is the contract, not the chat history.

Architecture decision records

Every irreversible call (schema, auth, third-party dependency, prod config) is written down, dated, and reviewed by the client's technical lead before code touches it.

Human-in-the-loop gates

Specific operations are blocked from auto-execution. The gate is not optional and not overridable by the agent. The list is short, named, and enforced by the harness.

Audit trail

Every agent decision has a trace. When the client asks "why does the PR look like this," I can replay the prompt, the tool calls, and the verification result, not improvise an answer.

These four items are unglamorous. They do not demo well. They are also the difference between a deliverable a client accepts and one a client litigates.

4. The marketing-vs-source meta-pattern

The bigger finding is structural. Eleven of the twelve frameworks have a visible gap between marketing claims and source reality. The pattern is consistent enough to be diagnostic.

FrameworkMarketing claimWhat the source actually shows
Claude-flow "Queen-led Byzantine-consensus Rust engine." queen is a JSON field literally {"term": 1}. "Byzantine consensus" is votes: Record<string, boolean> written to disk with writeFileSync. Rust is 20 of 1,339 source files, or 1.5% of the codebase.
BMAD "True scale-adaptive intelligence with G0-G6 gates." Four phase folders and a single user_skill_level config field. G0-G6 do not exist in the source. Search returns zero results.
GStack "10,000 LOC per week, 100 PRs per week." The project retracted this in its own CHANGELOG line 3282. docs/ON_THE_LOC_CONTROVERSY.md is 169 lines of self-critique. "9 role-based agents" is in fact 17-20 sprint-phase personas. "23 tools" is 47 SKILL.md files.
Superpowers "Enforces TDD. End of vibe coding." One SessionStart hook that injects prose. TDD is a prose request, not a gate. There is no static check, no CI hook, no failing-test enforcement.
Spec-kit "Vendor-agnostic." True at install time only. The CLI copies the same prompts into each agent's native skill folder. There is no runtime adapter and no per-agent tuning.
Traycer "93 minutes to a working dashboard." Searching site:traycer.ai for the 93-minute figure returns zero hits. The number appears in a third-party YouTube video, not on the vendor's site.
Paperclip "Autonomous CEO. Org-learning." ROADMAP.md shows the autonomy features as unstarted. The shipped code is a scheduler, a budget service, and a multi-vendor adapter.

Eleven out of twelve. The honest one is Spec-Kit, and even Spec-Kit overstates "vendor-agnostic." Two minutes of source reading beat two hours of marketing video, every time.

5. Why this happens (and why it is systemic)

Three mechanisms compound. None of them are conspiracies. All of them are predictable from how the ecosystem is structured.

The LLM-driven hype loop. The same models that write the README also write the docs, the announcement post, and the LinkedIn thread. The README inherits the language of the demo. The demo inherits the language of the prompt. The prompt inherits the language of a Twitter thread. By the time a human reads it, the file has been edited by an author who has every incentive to use the most ambitious verb in the chain. "Coordinates" becomes "orchestrates." "Writes JSON" becomes "achieves consensus." There is no editor in the loop.

Star-driven self-acceleration. A 50,000-star repo gets reviewed against its star count, not against its source. The reviewers assume the work has already been audited by 50,000 prior eyeballs. Stars travel faster than diffs. The most extreme claim travels fastest because it makes the best screenshot.

YouTube secondary amplification. A working harness gets a 20-minute video. The video author paraphrases the README in their own voice. The paraphrase gets cited back to the project as if the project itself made the claim. The "93-minute dashboard" line for Traycer is a YouTuber's number, not the vendor's, but it now circulates as the vendor's. Once a number is in the loop, it stays.

The defense against all three is the same. Open the source. Count the lines. Check the dates. Read what the project says about itself in CHANGELOG.md, ROADMAP.md, and the issue tracker. That work takes thirty minutes and will save weeks.

6. The six-layer FDE delivery pipeline

Here is what I would actually build for a client engagement. Six layers. Each layer steals one primitive from the twelve frameworks. No framework gets installed wholesale.

  1. Spec capture (Layer 1)Steal Spec-Kit's four markdown templates: spec, plan, tasks, constitution. Copy them. Do not install the CLI. The spec.md uses P1/P2/P3 user stories and goes in a pull request the client approves before any agent runs. HIL gate: client sign-off on the PR is non-negotiable.
  2. Plan and architecture (Layer 2)Use Claude Code's plan mode to draft, then commit the output as plan.md plus ADRs next to the spec. HIL gate: any schema change, auth change, third-party dependency, or prod config change requires the client's tech lead to sign off in writing.
  3. Execution (Layer 3)Steal GSD's worktree dispatch pattern from execute-phase.md lines 540-547. Add an intra-wave files_modified collision check: if two queued agents touch the same file, force them sequential. Claude Code's Task tool supports isolation="worktree" already. HIL gate: none inside a single task; the checkpoint is the wave boundary.
  4. Verification (Layer 4)Steal Archon's "deterministic node, not LLM judge" pattern. "Done" means typecheck && lint && test exit 0, not a sentinel string. LLM-judge is a supplementary signal, never primary. HIL gate: any verifier failure escalates. The agent is not allowed to modify the verifier to make it pass. That is the classic Goodhart trap (see BMAD issue #1332).
  5. Code review (Layer 5)Steal Superpowers' two-stage subagent review. One agent gets only the spec plus the diff and asks "what spec requirements are satisfied, what is missing." A second agent gets only the diff and asks DRY, YAGNI, security, error-handling questions. Then run a third pass with ChatGPT Codex for cross-vendor coverage of the same diff. HIL gate: a human reviews the diff before merge to main, even if all three AI reviewers approve.
  6. Cost and audit (Layer 6)Steal Paperclip's budget hard-stop from server/src/services/budgets.ts:958. Wire it as a hook that pauses all agents and cancels the queue when a per-feature, per-day, or per-engagement cap trips. Enable Claude Code's native OpenTelemetry export to Langfuse or Datadog. HIL gate: a hard-stop trip requires a human to unpause. The agent does not get to argue for more quota.

Five primitives stolen. Four rules and three skills written. Zero frameworks installed. Total assembly time under a day.

7. The ten never-auto list

These are non-negotiable. They do not become safer because the agent feels confident. They do not become safer because the test suite is green. They do not become safer because the model is new. They stay HIL forever.

  1. Merging to a protected branch (main, release).
  2. npm publish, package release, or any customer-facing deploy.
  3. Database schema migrations on production.
  4. Deleting anything (branch, table, record, customer data).
  5. Sending email or messages in the client's name.
  6. Changing customer-facing copy, pricing, or terms of service.
  7. Using a third-party API key, especially payment, SMS, or mass email.
  8. Any payment, trade, or fund movement.
  9. Any IAM, permission, or access-control change.
  10. Unpausing after a budget hard-stop trip.
The Ralph "overnight 90% completion" pattern is fine for a personal LeetCode bot. It is a liability disaster on a client engagement. The number of "wow that was fast" Ralph runs that produce a deletable mistake at 3 a.m. is non-zero. The point of HIL is not to slow the agent. It is to make sure the agent cannot create a category of damage that takes more time to undo than the time it claims to save.

8. What to install (nothing) and what to steal (five things)

The summary fits in one table.

#PatternStolen from
1Worktree-isolated parallel dispatch, with a same-file collision check that forces colliding tasks sequential before any agent spawns.GSD workflows/execute-phase.md:540-547.
2Budget hard-stop circuit breaker: per-task, per-day, per-engagement caps. Threshold crossed pauses agents and cancels the queue, waits for a human to unpause.Paperclip services/budgets.ts:958.
3External task table as control surface: long-running periodic work writes its state to a file the agent reads on next wake, instead of carrying state inside one conversation.Ralph prd.json, Archon YAML DAG.
4Markdown templates for spec, plan, tasks, and constitution. Copied into a project's specs/ folder. The CLI that wraps them is not installed.Spec-Kit templates/*-template.md.
5Two-stage subagent review: a spec-compliance reviewer reads spec plus diff, then a code-quality reviewer reads diff only. Each gets a fresh context. Both pass before a human looks.Superpowers subagent-driven-development/*-prompt.md.

The baseline of Claude Code plus ChatGPT Codex already covers multi-agent dispatch, plan mode, subagent isolation, MCP tooling, hooks, and headless runs. Adding another harness on top duplicates that surface. Adding five small primitives on top adds capability without adding ceremony.

That is the whole posture. The framework ecosystem is providing a small number of useful patterns inside a large amount of ceremony. The job is to extract the patterns and discard the ceremony.

One-sentence version

Twelve popular AI coding harnesses solve the easy half of the work, leaving spec sign-off, ADRs, HIL gates, and audit trail unclaimed, which is exactly the surface area where Forward-Deployed Engineering earns its money.
